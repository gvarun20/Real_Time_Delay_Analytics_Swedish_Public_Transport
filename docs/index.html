Why this project exists

Schedules and live delays were never meant to sit in separate folders forever.

Swedish transit publishes planned times (GTFS static) and live arrival updates (GTFS-RT). Analysts still need answers like: which routes run late, which stops are worst at rush hour, and how delays change by hour and weekday.

This project builds the missing production-style path: ingest on a schedule → join → model → quality-check → serve charts — at $0 cloud cost using Docker locally, with a public demo online.

Designed workflow

The pipeline, end to end

Every stage has a clear job. Raw files are kept immutable. Transforms write curated warehouse tables. The dashboard only reads — never writes.

  1. SOURCE · Trafiklab

    Open GTFS static + GTFS-RT TripUpdates

    Static Sweden feed + Sweden realtime feed for operator sl. Same ID family is required so joins match (documented in ADR-003).

  2. INGEST · Airflow

    gtfs_static_ingest + gtfs_realtime_ingest

    Daily static zip and 15-minute realtime snapshots land under data/raw/static/{date}/ and data/raw/realtime/{date}/{time}/ with metadata.json sidecars. Retries + failure callbacks included.

  3. TRANSFORM · PySpark

    gtfs_transform → jobs/transform_gtfs.py

    Filter SL trips, join stop times with TripUpdates, compute delay_seconds, upsert dimensions, load fact_trip_delay. Then run the data quality gate.

  4. WAREHOUSE · PostgreSQL

    Kimball star schema in transit_dw

    Grain: one fact row ≈ one trip × stop × service date × stop sequence. Tables: dim_date, dim_route, dim_stop, dim_vehicle_type, fact_trip_delay, pipeline_runs.

  5. QUALITY · validate_data_quality

    Fail the DAG on critical errors; warn on outliers

    Checks include non-empty facts, null trip IDs, delay bounds, null-delay rate, duplicate grain, and static file freshness. Results written to pipeline_runs.dq_status.

  6. SERVE · Streamlit

    Interactive analytics UI (public + local)

    Filters for date / route / vehicle type. KPIs, route bars, hour×weekday heatmap, stop map, worst-stops table, delay histogram. Public site uses a sample export; local mode reads live Postgres.

Data model

Star schema at the center

Kimball star schema = put measurements in a fact table and descriptive labels in dimension tables. Dimensions answer who / what / where / when. The fact stores scheduled time, actual time, and delay_seconds.

Why we use it: dashboard filters stay simple, every row has a clear meaning (“grain”), and this is the standard analytics model interviewers expect — much easier than querying raw GTFS files every time.

Full beginner guide on GitHub: 03-star-schema-explained.md. DDL: sql/schema.sql.

Real engineering, not a happy-path demo

Bugs we hit — and documented

Week 2 alone uncovered stacked failures. Each fix is written down so the next person (or an interviewer) can follow the reasoning.

  • Feed ID mismatch — Sweden static IDs ≠ regional realtime IDs → 0 join rows.
  • Wrong Postgres DB name in Docker env (transit vs transit_dw).
  • Silent RETURNING drop in execute_values without fetch=True.
  • CardinalityViolation from duplicate realtime keys on full loads.
  • Missing fs_default connection broke Airflow FileSensor.

Run it yourself

Clone → configure → Docker → dashboard

The public sites are for demos. The full Airflow + Spark + Postgres stack still runs on your machine. Full steps are in the README.

    Copy .env.example → .env

    Add Trafiklab keys. Keep STATIC_FEED and REALTIME_FEED in the same family.

    Bootstrap Docker

    .\scripts\bootstrap.ps1 — Airflow UI on :8081, Postgres on :5433.

    Trigger DAGs

    Ingest static + realtime, then gtfs_transform for a date with both feeds.

    Open local Streamlit

    py -m streamlit run dashboard/app.py against live Postgres.
