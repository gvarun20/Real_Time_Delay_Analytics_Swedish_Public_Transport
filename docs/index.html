Delay Pipeline

From raw GTFS feeds to delay insights — orchestrated, modeled, and shown in a live dashboard.

The problem

Schedules and live delays live in different files.

Swedish public transport publishes planned times (GTFS static) and live arrival updates (GTFS-RT). Alone, they do not easily answer: which routes are late, which stops hurt most at rush hour, or how delays change by hour and day.

  • Need reliable, scheduled downloads — not one-off manual pulls
  • Need a clean join of schedule ↔ realtime on trip and stop IDs
  • Need an analytical model analysts can query
  • Need a UI where you select filters and see results

The goal

A production-style pipeline at zero cloud cost.

Build the full path: ingest → transform → warehouse → quality gate → dashboard — runnable locally in Docker, with tests in CI, and a public project page for demos.

  • Apache Airflow
  • PySpark
  • PostgreSQL
  • Streamlit
  • Docker

Architecture

How data moves through the system

Think of it as a factory line: download → store raw files → transform → curated tables → charts.

  1. 01
    Trafiklab APIs Static GTFS zip + realtime TripUpdates for SL Stockholm.
  2. 02
    Airflow DAGs Scheduled ingest with retries; files land under data/raw/.
  3. 03
    PySpark transform Join schedule with live updates; compute delay_seconds.
  4. 04
    PostgreSQL star schema Dimensions (date, route, stop, vehicle type) + fact_trip_delay.
  5. 05
    Dashboard + quality checks Streamlit UI for KPIs, charts, map, and stops; DQ task fails on bad data.
